# Project Recyclability Audit Report

## 📋 Summary
* **Repository:** `tanDivina/IslandFlow-AWS`
* **Verdict:** 🟢 **ALLOWED**
* **Compatibility Score:** `75/100`

### 🔍 Verdict Reason
The official Qwen Cloud Hackathon rules explicitly allow projects that existed prior to the hackathon, provided they are **"significantly updated after the start of the Hackathon Submission Period"** (May 26, 2026). Our audit shows that by converting this concierge booking template into an advanced, production-grade agent with persistent cross-session memory (Track 1) and integrating Alibaba Cloud services (OSS), we satisfy all technical requirements and eligibility policies.

---

## 🛠️ Project Profile
* **Project Summary:** An autonomous AI concierge and logistics dispatcher designed for eco-lodges, hotels, and activity operators. It handles scheduling, real-time weather monitoring, interactive proposal swaps (human-in-the-loop), and transaction state management.
* **Detected Tech Stack:** React, Vite, FastAPI (Python), FastMCP, MongoDB / Mock JSON DB, Google GenAI SDK (ADK), and standard OpenAI compatible endpoints for Qwen (DashScope API).

---

## 📜 Hackathon Policy & Alignment
* **Rules Quote/Policy:** *"Projects must be either newly created by the Entrant or, if the Entrant’s Project existed prior to the Hackathon Submission Period, must have been significantly updated after the start of the Hackathon Submission Period. Entrants should explain how their Project was significantly updated during the Submission Period."*

### 💪 Strengths & Matches
- **Exquisite UX & Multi-Agent Flow:** Includes scrolling MCP tool logs, timeline itineraries, weather shifting controls, and interactive chat proposal cards.
- **Qwen-Ready Core:** Features a pre-built OpenAI-compatible `run_qwen_agent` loop using DashScope inside `backend/agent.py`.
- **Open Source Compliant:** Contains an active detectable `LICENSE` file (MIT) at the repository root.

### ⚠️ Gaps & Missing Requirements
- **No Alibaba Cloud API Proof:** Currently uses AWS branding (`IslandFlow-AWS`) and lacks code demonstrating direct interaction with Alibaba Cloud APIs.
- **Generic Memory Layer:** The current profile update logic is simple and lacks deep cross-session conversational summaries, which are critical for **Track 1: MemoryAgent**.
- **Default LLM Provider:** The template defaults to Gemini (`LLM_PROVIDER=gemini`) instead of Qwen.

---

## 🗺️ Required Adjustments Plan

### 1. Integrate Alibaba Cloud Object Storage Service (OSS) (Effort: **Medium**)
* **Description:** The rules require "Proof of Alibaba Cloud Deployment: Proof must be a link to a code file in their code repo that demonstrates use of Alibaba Cloud services and APIs."
* **Instructions:**
  1. Install the Alibaba Cloud OSS Python SDK (`oss2`) via `backend/requirements.txt`.
  2. Implement an exporter in `backend/db.py` or a new `backend/ali_oss.py` module that uploads printed itineraries or travel invoices (generated by `generate_itinerary`) to an Alibaba Cloud OSS bucket.
  3. Return a public OSS link to the frontend so guests can download/print their official itineraries directly from Alibaba Cloud.
  4. Document the OSS code link clearly in our README as submission proof.

### 2. Implement Deep Cross-Session Persistent Memory (Track 1: MemoryAgent) (Effort: **High**)
* **Description:** To win the **MemoryAgent** track, the agent must autonomously accumulate experience, remember preferences, and handle multi-turn, cross-session interactions.
* **Instructions:**
  1. Add a `conversational_memories` collection in `db.py` to store key insights and preferences extracted from user conversations.
  2. Write an autonomous memory extractor tool `extract_and_save_memory(guest_id, context)` that Qwen can trigger asynchronously after a user interaction to store key preferences (e.g. *"Alex loves eco-tours", "Allergic to nuts"*).
  3. Update `run_qwen_agent` to load these memory logs and user preferences as part of the background context during initialization, ensuring the agent remembers the user across different sessions.

### 3. Change Default LLM Provider to Qwen (Effort: **Low**)
* **Description:** Ensure that out of the box, the application runs on the Qwen models via DashScope.
* **Instructions:**
  1. Update `.env.example` and default settings in `backend/agent.py` to make `LLM_PROVIDER=qwen` the default.
  2. Modify the system prompt to explicitly reference Qwen-based instructions and verify tool schema compliance with DashScope constraints.

### 4. Apply Hero-Apps Style Guide & Premium Dark Glassmorphism Theme (Effort: **Medium**)
* **Description:** Transform the styling of the application to follow Dorien's premium dark glassmorphism rules.
* **Instructions:**
  1. Edit `frontend/src/index.css` to implement a dark ocean theme: pitch black backdrops (`#050507`), deep charcoal panel containers (`#0c0c0f`), semi-transparent borders (`rgba(255, 255, 255, 0.06)`), and neon-green primary accents (`#a8ff35` & `#7cd315`).
  2. Ensure there are no solid box outlines around branding.
  3. Ensure all icons are premium custom inline SVGs.
  4. Cache-bust stylesheets by appending version tags in references (e.g. `index.css?v=1.0.1`).

## 🎯 Recommended Strategic Approach
We should submit to **Track 1: MemoryAgent**. By implementing the cross-session memory module in MongoDB/MockDB and integrating Alibaba Cloud OSS to host and serve receipts, we resolve the two largest requirements of the hackathon:
1. Creating a high-impact, memory-driven agent project.
2. Furnishing verifiable in-repo proof of using Alibaba Cloud services/APIs.
This converts our existing project into a premium, rule-compliant submission with maximum competitive potential.
